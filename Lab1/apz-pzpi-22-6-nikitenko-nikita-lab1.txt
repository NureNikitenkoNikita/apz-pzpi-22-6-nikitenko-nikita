	2 ЗАВДАННЯ
Потрібно розробити клієнтську частину програмної системи (мобільний додаток) та інтегрувати її з існуючою серверною частиною, що забезпечує моніторинг лих, управління даними та взаємодію з IoT-пристроями.
Основні цілі:
1. Розробка мобільного клієнта: Створення функціонального мобільного додатка на базі React Native для платформи Android.
2. Інтеграція з бекендом: Забезпечення коректної взаємодії мобільного додатка з RESTful API серверної частини для авторизації, отримання даних про лиха та реального часу.
3. Візуалізація даних: Відображення географічних даних про лиха на інтерактивній карті (з використанням Leaflet/OpenStreetMap).
4. Реалізація користувацьких сценаріїв: Впровадження функціоналу входу/виходу, перегляду лих та їх деталей.
5. Документування: Оформлення звіту з описом виконаної роботи, UML діаграмами та специфікацією вимог до програмного забезпечення (SRS).

	1. Business requirements 
1.1 Background
Природні катастрофи, такі як землетруси, повені, лісові пожежі та виверження вулканів, загрожують життю людей і спричиняють значні економічні збитки. У сучасному світі критично важливо мати оперативну систему моніторингу, яка дозволяє в реальному часі відстежувати події, аналізувати їх та інформувати користувачів для зменшення наслідків катастроф. 
1.2 Business Opportunities  
Розроблена програмна система дозволяє:
1. Оперативно відстежувати природні катастрофи у режимі реального часу. 
2. Інформувати населення про загрози. 
3. Надавати інструменти для історії катастроф
Основні бізнес-цілі: 
* Забезпечити стабільну роботу системи моніторингу з високою точністю 
даних. 


* Зменшити час реакції на критичні події. 


* Надати користувачам зручний інтерфейс для перегляду інформації про 
катастрофи. 




Потреби ринку та клієнтів: 


* Наявність інтегрованої платформи для моніторингу природних 
катастроф. 
* Швидке та точне відображення подій на інтерактивній карті. 
* Інформування про критичні події через візуальні індикатори, графіки та 
сповіщення. 
* Доступ до детальної статистики для аналізу.
2. Vision of the solution 
2.1 Vision Statement 
Disaster Monitor  – це інноваційна глобальна система моніторингу природних катастроф, що забезпечує оперативне відстеження, аналіз та інформування про критичні події у режимі реального часу. 
2.2 Major Features 
MF-1: Авторизація та управління доступом
Ця функція забезпечує безпечний вхід для користувачів та розмежування доступу на основі ролей. Це означає, що адміністратори, модератори та звичайні користувачі матимуть різні рівні дозволів, захищаючи конфіденційні дані та функції.
MF-2: Збір даних
Система автоматично отримує дані про лиха від підключених IoT-пристроїв. Це дозволяє швидко та точно збирати інформацію про події, такі як пожежі чи повені, без ручного вводу.
MF-3: Зберігання та управління даними
Буде створено надійну базу даних для зберігання всієї інформації про лиха (місце, час, тип) та дані користувачів. Це гарантує цілісність, доступність та можливість ефективного пошуку даних.
MF-4: Візуалізація даних
Дані про лиха відображаються на інтерактивній географічній карті. Користувачі зможуть візуально оцінювати ситуацію, застосовувати фільтри та отримувати детальні дані, клацнувши на об'єкти на карті.


MF-5: Моніторинг у реальному часі
Система забезпечує автоматичне оновлення інформації про природні лиха без перезавантаження сторінки. Це гарантує, що користувачі завжди бачать найактуальніші дані, дозволяючи оперативно реагувати на зміни.


2.3 Assumptions and Dependencies 
* Обмеження:
   * Мобільний додаток розробляється лише для платформи Android.
   * Використання Leaflet.js та OpenStreetMap для карт, виключаючи інтеграцію з Google Maps SDK.
* Припущення:
   * Бекенд-сервер постійно доступний.
   * IoT-пристрої коректно функціонують та надсилають дані у потрібному форматі.
   * Користувачі мають стабільне інтернет-з'єднання.
3. Scope and Limitations 
3.1 Scope of Initial Release 
3.1.1 Server-side 


Сервер є центральним елементом системи, відповідальним за збір, обробку та аналіз даних про надзвичайні події.

* Збір даних: Сервер отримує актуальну інформацію про стихійні лиха (землетруси, повені, вулкани, пожежі)
* Фільтрація та аналіз: Отримані дані фільтруються за вибраним періодом (наприклад, 24 години, 3, 7 днів) та аналізуються на відповідність критичним рівням загрози (наприклад, поріг магнітуди для землетрусів).
* Оновлення даних: Інформація оновлюється автоматично



3.1.2 IoT-client


"IoT-клієнти" – великі глобальні та регіональні системи збору даних, що симулюють інформацію.


3.1.3 Web-client

Веб-клієнт – це основний інтерфейс для моніторингу катастроф, доступний через браузер.

* Інтерактивна карта: Відображає дані про надзвичайні події, використовуючи картографічні сервіси (OpenStreetMap) та бібліотеки для інтерактивного відображення.
* Функціональність: Включає фільтрацію подій, статистичний аналіз з візуалізацією та систему критичних сповіщень.
* Оновлення: Дані на карті оновлюються в реальному часі (наприклад, кожні 5 хвилин), забезпечуючи актуальний моніторинг.


3.1.4 Mobile client
Мобільний клієнт розширює доступ до ключових функцій системи, оптимізуючи їх для портативних пристроїв. Він надає:
* Перегляд карти катастроф у реальному часі.
* Push-сповіщення про критичні події, особливо релевантні для поточної або обраної геолокації користувача.
* Базові можливості фільтрації подій та налаштування сповіщень.
* Зручний інтерфейс, спеціально розроблений для мобільних пристроїв.


3.3 Limitations and Exclusions 
Система вимагає постійного підключення до інтернету, оскільки офлайн-доступ до даних не передбачено. Інтерфейс оптимізований переважно для десктопних браузерів; хоча мобільна веб-версія доступна, її функціональність може бути обмеженою порівняно з нативним мобільним додатком.




4. Business Context 
4.1 Operating Environment 


Операційне середовище:
1. Час доступу: Система повинна бути доступною 24/7, оскільки 
надзвичайні події можуть траплятися у будь-який час. 
3. Інфраструктура: Дані генеруються в випадковому порядку, адже реалізація за допомогою API не передбачена у проекті.
4. Продуктивність: Система повинна забезпечувати швидке завантаження 
карти, оновлення даних кожні 5 хвилин, а також швидке реагування на 
дії користувача. 
5. Безпека: Забезпечення захисту даних користувача, зокрема інформації 
про ІР та геолокацію, відповідно до сучасних стандартів безпеки. 
6. Сумісність: Підтримка сучасних веб-браузерів (Chrome, Edge, Brave) з 
адаптивним дизайном для різних пристроїв. Мобільні клієнти будуть 
розроблятися для поширених версій Android.
